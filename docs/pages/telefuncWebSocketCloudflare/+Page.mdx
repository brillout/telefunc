**Environment**: server.

Cloudflare-specific setup, architecture, and delivery guarantees for Telefunc channels and pub/sub.


## Setup

```ts
// worker.ts
// Environment: server

import { telefunc } from 'telefunc/cloudflare'

const tf = telefunc()

export const TelefuncDurableObject = tf.TelefuncDurableObject

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    const response = await tf.serve({ request, env, ctx })
    if (response) return response

    return new Response('Not found', { status: 404 })
  },
}
```

```jsonc
// wrangler.jsonc
{
  "compatibility_flags": ["nodejs_als"],
  "durable_objects": {
    "bindings": [
      { "name": "TelefuncDurableObject", "class_name": "TelefuncDurableObject" }
    ]
  },
  "kv_namespaces": [
    { "binding": "TelefuncKV", "id": "your-kv-namespace-id" }
  ],
  "migrations": [
    { "tag": "v1", "new_classes": ["TelefuncDurableObject"] }
  ]
}
```

> `compatibility_flags: ["nodejs_als"]` is required — Telefunc uses `AsyncLocalStorage` for request context.

## Architecture

Telefunc uses [Durable Objects](https://developers.cloudflare.com/durable-objects/) for channel state and pub/sub coordination. Channel state is in-memory JavaScript (closures, callbacks, local variables), so it must live on the same Durable Object instance as the WebSocket connection.

[KV](https://developers.cloudflare.com/kv/) stores two things:
- **Shard routing tokens** — pins a browser session to the same Durable Object across requests.
- **Pub/sub presence** — tracks which Durable Objects have active subscribers for each key.

### Buckets

A **bucket** is one of six geographic regions. Each bucket runs its own set of Durable Object instances so that channel state stays close to the users it serves:

| Bucket | Region |
|---|---|
| `wnam` | West North America |
| `enam` | East North America |
| `weur` | West Europe |
| `eeur` | East Europe |
| `apac` | Asia Pacific |
| `oc` | Oceania |

Telefunc maps each incoming request to a bucket using the Cloudflare colocation point (`request.cf.colo`), then continent (`request.cf.continent`), then `locationFallback`.

### Sticky routing

Every telefunction call and channel message from a browser must reach the **same session shard** — otherwise the Durable Object holding the channel state wouldn't see it.

On the first request, Telefunc resolves the shard from the bucket and scale config, stores a routing token in KV (TTL: 24 hours), and returns it to the client via the `x-telefunc-session` response header. The token includes the shard name so you can inspect which region and shard a session is pinned to (e.g. `telefunc-shard-weur-0:a1b2c3d4-...`). Subsequent requests send this token back so Telefunc can look up the same shard.

> **CDN / reverse proxy note**: If you place a CDN or reverse proxy in front of your Worker, make sure `x-telefunc-session` is not stripped from responses or requests. Telefunc uses this header to maintain session affinity — without it, each request would be routed to a random shard and channel state would be lost.

### Durable Object types

Within a bucket, there are two kinds of Durable Object:

- **Session shards** hold WebSocket connections and channel state. Browser sessions are pinned to one shard via the KV routing token.
- **Bucket coordinators** aggregate pub/sub fanout within a region so the authority doesn't need to address every session shard individually. Each channel key is deterministically hashed to one coordinator per bucket.


## Scaling

By default, Telefunc creates one session shard per bucket. Increase capacity with `scale`:

```ts
const tf = telefunc({ scale: 4 })
```

This creates 4 session shards and 2 bucket coordinators (`ceil(4 / 2)`) per bucket.

### Per-region scale

```ts
const tf = telefunc({
  scale: { weur: 5, enam: 3, apac: 2 },
})
```

Only specified regions get session shards. Unspecified regions have no Durable Object instances — requests from those regions fall back via `locationFallback`.


## Distributed pub/sub

Keyed channels (<Link href="/createChannel" />) use a three-tier fanout architecture to broadcast across edge locations.

### Publish flow

```
Publisher (session shard)
  → Authority DO (one per key — sequences and timestamps)
    → Bucket coordinators (one or more per active region)
      → Session shards (deliver to local subscribers)
```

1. The publisher's session shard forwards to the **authority** Durable Object for the key.
2. The authority serializes publishes through an exclusive dispatch chain, assigns a monotonic sequence number (starting from 1), reads active presence from KV, and selectively forwards to each **bucket coordinator** in regions with active subscribers.
3. Each bucket coordinator fans out to the **session shards** in its region.
4. Each session shard delivers to its local subscribers (including the sender when `selfDelivery` is enabled — the default).

The authority is a single point of serialization — it sees every publish for a key and assigns a globally monotonic `seq`. Bucket coordinators exist so this one authority only needs to send `O(buckets)` messages instead of `O(session shards)`.

### Ordering

Publishes for a given key are serialized through a single authority instance. The authority assigns a monotonic sequence number (`seq`) that increases by 1 for each publish, then fans out to bucket coordinators in order. Subscribers receive messages in publish order.

```ts
const receipt = await channel.publish({ user: 'Alice', text: 'Hello!', ts: Date.now() })
receipt.key  // 'chat:lobby'
receipt.seq  // monotonic sequence number for this key
receipt.ts   // server timestamp
```

### Presence

Telefunc tracks which Durable Objects have active subscribers using KV presence entries:

| Setting | Value |
|---|---|
| TTL | 90 seconds |
| Refresh interval | 30 seconds |

Presence entries are created when a channel subscribes to a key and deleted when it unsubscribes. The refresh timer keeps the entry alive as long as the channel is open. If a Durable Object is evicted by the runtime without unsubscribing (e.g. during a deployment), the entry expires after 90 seconds and the bucket is no longer included in fanout.


## Delivery guarantees

### Channel messages (`send` / `listen`)

| Guarantee | Details |
|---|---|
| Ordering | Messages within a channel are delivered in order. |
| Buffering | The server buffers messages while the client is disconnected, up to `config.channel.bufferLimit` (default: 512 KB). |
| Replay | Both sides keep a bounded replay buffer. On reconnect, missing frames are replayed and duplicates are ignored. |
| Acknowledgements | `send(data, { ack: true })` returns a `Promise` that resolves when the other side processes the message. |
| Loss | If the disconnect exceeds `config.channel.reconnectTimeout`, the channel closes with `ChannelNetworkError`. Older frames evicted from the replay buffer before reconnect are lost. |

### Pub/sub messages (`publish` / `subscribe`)

| Guarantee | Details |
|---|---|
| Ordering | Publishes for a given key are serialized through a single authority and delivered in order. |
| Delivery | `publish()` resolves after all active session shards have received the message. This confirms server-side fanout completed — delivery to the client happens over the channel wire and is not awaited. |
| Presence lag | Up to 90 seconds. An evicted Durable Object may still receive fanout attempts until its KV presence entry expires. A new subscriber may miss publishes from other publishers until its KV presence entry is written (typically a few milliseconds). |

> Once a pub/sub message reaches a session shard, it participates in the same replay and recovery semantics as regular channel messages — buffered if the client is disconnected, replayed on reconnect.


## Configuration

```ts
const tf = telefunc({
  // Durable Object binding name (default: 'TelefuncDurableObject')
  bindingName: 'TelefuncDurableObject',

  // KV binding name for shard tokens and pub/sub presence (default: 'TelefuncKV')
  kvBindingName: 'TelefuncKV',

  // Instance base name for Durable Objects (default: 'telefunc')
  instanceName: 'telefunc',

  // Session shards per location bucket (default: 1)
  scale: 1,

  // Fallback location when the request can't be mapped (default: 'weur')
  locationFallback: 'weur',

  // Restrict Durable Objects to a jurisdiction
  jurisdiction: 'eu',
})
```

| Option | Default | Description |
|---|---|---|
| `bindingName` | `'TelefuncDurableObject'` | Durable Object binding name in `wrangler.jsonc`. |
| `kvBindingName` | `'TelefuncKV'` | KV namespace binding name. |
| `instanceName` | `'telefunc'` | Base name for Durable Object instance IDs. |
| `scale` | `1` | Session shards per bucket. Number or per-bucket map. |
| `locationFallback` | `'weur'` | Canonical bucket for unmapped requests. |
| `jurisdiction` | — | Cloudflare [jurisdiction](https://developers.cloudflare.com/durable-objects/reference/data-location/#restrict-durable-objects-to-a-jurisdiction) such as `'eu'` or `'fedramp'`. |


## Hibernation

> **[Durable Object hibernation](https://developers.cloudflare.com/durable-objects/best-practices/websockets/#why-use-hibernation) while channels are open is not supported.**
>
> Channels are plain in-memory JavaScript: closures, local variables, callbacks. If the Durable Object hibernates, that heap state is gone. Telefunc keeps the Durable Object alive for the lifetime of each open channel.

Once all channels close and no clients are connected, the Durable Object can hibernate again — after the reconnect and idle windows expire.

Shorter windows let the Durable Object hibernate sooner but reduce tolerance for disconnects:

```ts
import { config } from 'telefunc'

config.channel = {
  reconnectTimeout: 5_000,
  idleTimeout: 5_000,
  pingInterval: 2_000,
}
```
