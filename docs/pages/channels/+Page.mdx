Telefunc supports persistent, bidirectional communication channels over WebSocket using <Link text={<code>createChannel()</code>} href="/createChannel" />.

> Channels are useful for real-time features like chat, live dashboards, or any scenario where the server and client need to exchange messages continuously.


## Example

```ts
// Chat.telefunc.ts
// Environment: server

import { createChannel } from 'telefunc'

export async function onChat() {
  const channel = createChannel<string, string>()

  channel.listen((msg) => {
    console.log('from client:', msg)
    channel.send(`echo: ${msg}`)
  })

  channel.onClose(() => {
    console.log('channel closed')
  })

  return { channel: channel.client }
}
```

```tsx
// Chat.tsx
// Environment: client

import { onChat } from './Chat.telefunc'

const { channel } = await onChat()

channel.listen((msg) => {
  console.log('from server:', msg) // 'echo: hello'
})

channel.send('hello')
```

Return `channel.client` to the client — it transparently serializes into a `ClientChannel` connected over WebSocket.


## Channel interface

Both server and client channels share the same interface:

| Method | Description |
|---|---|
| `send(data)` | Send a message |
| `sendBinary(data)` | Send raw bytes (`Uint8Array`) |
| `listen(cb)` | Receive messages |
| `listenBinary(cb)` | Receive raw bytes |
| `onOpen(cb)` | Fires when the channel is connected |
| `onClose(cb)` | Fires when the channel closes |
| `close()` | Close the channel |
| `isClosed` | Whether the channel has been closed |


## Cancellation

Close a channel from the client:

```ts
channel.close()
```

You can also close all channels returned by a telefunction using <Link text={<code>abort()</code>} href="/cancellation#abort" />:

```ts
import { abort } from 'telefunc/client'

const result = await onInit()
// Closes all channels in the result
abort(result)
```

> `abort()` calls `close()` on each channel internally — the effect is the same. `abort()` is useful when a telefunction returns multiple channels or a mix of channels and streams.


## WebSocket setup

Channels require a WebSocket connection. In Vite dev mode, this is handled automatically.

For production, install the WebSocket adapter for your runtime — see <Link href="/telefuncWebSocket" />.

> All channels — and streaming values when using <Link text={<code>transport: 'ws'</code>} href="/transport" /> — share a single multiplexed WebSocket connection per server URL.


## Reconnection

If the WebSocket connection drops (e.g. network issue), the client automatically reconnects with exponential backoff. Open channels are reconciled — both sides replay missed frames — so the channel resumes without data loss.

> Reconnection is transparent: `onClose` and `onOpen` don't fire during a reconnect. They only fire when the channel is permanently closed or initially opened.

If the client can't reconnect within 60 seconds, all channels are closed permanently.


## TypeScript

`createChannel<TSend, TReceive>()` is generic:
- `TSend` — type of messages the server **sends**
- `TReceive` — type of messages the server **receives**

```ts
// Environment: server

import { createChannel } from 'telefunc'

type ServerMessage = { type: 'tick'; count: number }
type ClientMessage = { type: 'ping' }

export async function onInit() {
  const channel = createChannel<ServerMessage, ClientMessage>()

  channel.send({ type: 'tick', count: 1 })
  channel.listen((msg) => {
    msg.type // 'ping'
  })

  return { channel: channel.client }
}
```

The client channel has the types **flipped** — it sends `ClientMessage` and receives `ServerMessage`:

```ts
// Environment: client

const { channel } = await onInit()

channel.send({ type: 'ping' })
channel.listen((msg) => {
  console.log(msg.count) // number
})
```


## Cross-instance pub/sub

> **Feedback welcome** — built-in pub/sub is not yet implemented. If this is blocking you, please open an issue.

When you run multiple server instances (e.g. <Link text="Cloudflare shards" href="/telefuncWebSocket#scaling-with-shards" /> or multiple Node processes behind a load balancer), channels on different instances cannot directly message each other. Each channel lives in the process that created it.

You can implement cross-instance pub/sub today by wiring channels to a shared message bus (Redis, Cloudflare Durable Objects, etc.) yourself:

```ts
// Environment: server

import { createChannel } from 'telefunc'

export async function onJoinRoom(roomId: string) {
  const channel = createChannel<string, string>()

  // Subscribe this channel to the shared topic
  const unsubscribe = pubSub.subscribe(roomId, (msg) => channel.send(msg))

  // Forward client messages to the shared topic
  channel.listen((msg) => pubSub.publish(roomId, msg))

  channel.onClose(unsubscribe)

  return { channel: channel.client }
}
```

In the future, Telefunc plans to expose built-in `channel.publish(topic, msg)` / `channel.subscribe(topic, cb)` APIs that handle cross-instance routing automatically — using a central Durable Object on Cloudflare and pluggable adapters (Redis, etc.) elsewhere.
