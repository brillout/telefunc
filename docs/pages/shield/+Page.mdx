**Environment**: server.

You use `shield()` to guarantee the type of telefunction arguments. (As explained in <Link href="/RPC" />,
telefunctions are public and need protection.)

```js
// CreateTodo.telefunc.js
// Environment: server

export { onNewTodo }

import { shield } from 'telefunc'
const t = shield.type

shield(onNewTodo, [t.string])
async function onNewTodo(text) {
  // `text` is guaranteed to be a `string`: if `onNewTodo(42)` is called then Telefunc
  // throws an error that `text` should be a `string` (instead of a `number`)
}
```

> If you use TypeScript, then Telefunc automatically defines `shield()`, see <Link href="#typescript-automatic" />.


## TypeScript - Automatic

If you use TypeScript, then Telefunc automatically generates `shield()` for each telefunction.

In other words: telefunction argument types are automatically validated at runtime:

```ts ts-only
// hello.telefunc.ts

// No need to define a shield() when using TypeScript: Telefunc automatically generates
// it. For example here, Telefunc automatically aborts the telefunction call if the
// argument is `hello({ name: 42 })` and throws an error that `name` should be a `number`.
export async function hello({ name }: { name: string }) {
  return `Welcome to Telefunc, ${name}.`
}
```

With Telefunc, not only can you seamlessly re-use types across your frontend and backend code, but you also get automatic type-safety at runtime. If you use a TypeScript ORM (e.g. [Prisma](https://www.prisma.io/)) or SQL builder (e.g. [Kysely](https://github.com/koskimas/kysely) and [others](https://github.com/stars/brillout/lists/sql)), then you get end-to-end type safety all the way from database to frontend.

> For a faster development, Telefunc doesn't generate `shield()` and your telefunction arguments aren't validated during development.
> Telefunc only generates `shield()` upon building your app for production.
> You can enable the generation of `shield()` for development by setting <Link href="/shield-config">`config.shield.dev`</Link> to `true`.

> Telefunc's automatic `shield()` generation only works for stacks that transpile server-side code (Next.js, Vite, Vike, SvelteKit, Nuxt, etc.).
>
> For stacks that don't transpile server-side code (e.g. React Native and Parcel), you need to define `shield()` manually: see <Link href="#typescript-manual" />.


## TypeScript - Manual

If you define `shield()` manually (instead of using Telefunc's automatic `shield()` generator as described in <Link href="#typescript-automatic" />), then use Telefunc's decorator-style API:

```ts ts-only
import { shield } from 'telefunc'

shield(onNewTodo, [shield.type.string])
export async function onNewTodo(text: string) {
  // ✅ TypeScript knows that `text` is of type `string`
}
```

Automatic TypeScript `shield()` generation remains the default and most transparent option. Manual `shield()` is mainly useful for JavaScript users, for stacks that don't transpile server-side code, or when you want Telefunc to execute a specific runtime validator.

If you write `type Todo = z.infer<typeof TodoSchema>` and then `export async function onNewTodo(todo: Todo)`, automatic `shield()` generation can use `Todo` directly. The `z.infer` pattern works fine here. Telefunc reads the telefunction's resulting TypeScript signature and generates a `shield()` from that type.

That means the TypeScript signature alone is enough whenever the runtime contract you want Telefunc to enforce is fully described by the resulting TypeScript type. In that case, you can rely on automatic `shield()` generation and you don't need to pass `TodoSchema` manually.

Passing `shield(onNewTodo, [TodoSchema])` adds something extra: Telefunc runs the schema itself at runtime. This is useful when you want schema-specific runtime behavior, such as validator-defined checks or the validator's own issue details, instead of only the shape that TypeScript exposes after inference.

Concretely, consider these two cases:
- if your schema is `z.object({ text: z.string() })`, then `type Todo = z.infer<typeof TodoSchema>` and `shield(onNewTodo, [TodoSchema])` effectively describe the same runtime shape for Telefunc's purposes
- if your schema is `z.object({ text: z.string().min(1) })`, then automatic TypeScript `shield()` generation still sees `text: string`, while manually passing `TodoSchema` also runs Zod's `.min(1)` check

The same clarification applies if the value comes from Zustand on the client. Zustand itself doesn't change how Telefunc validates inputs; it only changes where the client-side value is stored. For example, if a Zustand store contains `draftTodo: Todo` and you call `onNewTodo(useTodoStore.getState().draftTodo)`, then automatic TypeScript `shield()` generation validates the incoming argument from the telefunction's `Todo` type. If you instead want Telefunc to run `TodoSchema` itself for that Zustand value, then pass `TodoSchema` manually with `shield(onNewTodo, [TodoSchema])`.

In summary:
- automatic TypeScript `shield()` generation is enough when the telefunction's TypeScript signature already expresses the full contract you want Telefunc to validate at runtime
- manually passing a Zod-like or Standard Schema validator adds that validator's own runtime checks and issue details
- Zustand is orthogonal here: whether the client value comes from Zustand, React state, a form library, or any other source, Telefunc still validates according to the telefunction signature unless you explicitly give Telefunc a runtime schema to execute

For example, you can bring a Zod-like or <a href="https://standardschema.dev" target="_blank">Standard Schema</a>-compatible validator without an extra wrapper:

```ts ts-only
import { shield } from 'telefunc'
import { z } from 'zod'

const TodoSchema = z.object({ text: z.string() })
type Todo = z.infer<typeof TodoSchema>

shield(onNewTodo, [TodoSchema])
export async function onNewTodo(todo: Todo) {
  // `todo` is validated at runtime by Zod.
  // If validation fails, Telefunc rejects the call with the schema's message.
}
```

On the client, Telefunc throws a `ValidationError` with an `.issues` array for shield validation failures. This is separate from `Abort()`: shield validation still rejects the call with HTTP 422, while `Abort()` remains the way to intentionally abort a telefunction call.

This also keeps Telefunc's validation philosophy consistent:
- use `shield()` (or automatic TypeScript `shield()` generation) for transport-level contract validation such as wrong input types or malformed calls
- return issues from your telefunction for business/domain validation

In other words: `ValidationError` is for callers breaking the telefunction contract before your telefunction runs, while returned issues are for expected application-level validation once your telefunction logic starts.


## Common types

Examples showcasing the most common `shield()` types:

```js
// TodoList.telefunc.js
// Environment: server

import { shield } from 'telefunc'
const t = shield.type

shield(onTextChange, [t.number, t.string])
async function onTextChange(id, text) {
  // typeof id === 'number'
  // typeof text === 'string'
}

shield(onCompletedToggle, [{ id: t.number, isCompleted: t.boolean }])
async function onCompletedToggle({ id, isCompleted }) {
  // typeof id === 'number'
  // typeof isCompleted === 'boolean'
}

shield(onTagListChange, [t.array(t.string)])
async function onTagListChange(tagList) {
  // tagList.every(tagName => typeof tagName === 'string')
}

shield(onNewMilestone, [{
  name: t.string,
  deadline: t.nullable(t.date),
  ownerId: t.optional(t.number)
}])
async function onNewMilestone({ name, deadline, ownerId }) {
  // typeof name === 'string'
  // deadline === null || deadline.constructor === Date
  // ownerId === undefined || typeof ownerId === 'number'
}

shield(onStatusChange, [t.or(
  t.const('DONE'),
  t.const('PROGRESS'),
  t.const('POSTPONED')
)])
async function onStatusChange(status) {
  // status === 'DONE' || status === 'PROGRESS' || status === 'POSTPONED'
}
```


## All types

List of `shield()` types:

`const t = shield.type` | TypeScript | JavaScript
-|-|-
`t.string` | `string` | `typeof value === 'string'`
`t.number` | `number` | `typeof value === 'number'`
`t.boolean` | `boolean` | <code>value === true &#x7c;&#x7c; value === false</code>
`t.date` | `Date` | `value.constructor === Date`
`t.array(T)` | `T[]` | `value.every(element => isT(element))`
`t.object(T)` | `Record<string, T>` | `Object.values(value).every(v => isT(v))`
`{ k1: T1, k2: T2, ... }` | <code>{'{ k1: T1, k2: T2, ... }'}</code> | <code>isT1(value.k1) && isT2(value.k2) && ...</code>
`t.or(T1, T2, ...)` | <code>T1 &#x7c; T2 &#x7c; ...</code> | <code>isT1(value) &#x7c;&#x7c; isT2(value) &#x7c;&#x7c; ...</code>
`t.tuple(T1, T2, ...)` | `[T1, T2, ...]` | `isT1(value[0]) && isT2(value[1]) && ...`
`t.const(val)` | `val as const` | `value === val`
`t.optional(T)` | <code>T &#x7c; undefined</code> | <code>isT(value) &#x7c;&#x7c; value === undefined</code>
`t.nullable(T)` | <code>T &#x7c; null</code> | <code>isT(value) &#x7c;&#x7c; value === null</code>
`t.any` | `any` | `true`
