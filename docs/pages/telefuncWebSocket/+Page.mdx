**Environment**: server.

Install the Telefunc WebSocket upgrade handler on your server. This is what powers <Link text={<code>createChannel()</code>} href="/createChannel" />, returned function proxies, and WebSocket transport.

## When do you need this?

You need <Link text={<code>telefuncWebSocket()</code>} href="/telefuncWebSocket" /> in production if you use any of the following:

- <Link text={<code>createChannel()</code>} href="/createChannel" />
- <Link href="/real-time#function-passing">Function passing</Link>
- <Link href="/transport">WebSocket transport</Link> for <Link href="/stream">streamed values</Link>

You do not need it for plain telefunction calls, or for <Link href="/stream">streamed values</Link> sent over the default HTTP transport or SSE.

Pick the adapter for your runtime:

## Node.js

```ts
import { telefuncWebSocket } from 'telefunc/websocket/node'
import { createServer } from 'node:http'

const server = createServer()
const ws = telefuncWebSocket()
ws.install(server)
```

Works with any Node.js HTTP server such as Express, Fastify, or Hono with `@hono/node-server`.

`install()` is idempotent, so calling it more than once on the same server is safe.

## Bun

```ts
import { telefuncWebSocket } from 'telefunc/websocket/bun'

const ws = telefuncWebSocket()

Bun.serve({
  port: 3000,
  websocket: ws.websocket,
  fetch(request, server) {
    const upgraded = ws.handleUpgrade(request, server)
    if (upgraded) return upgraded
    // ... rest of your fetch handler
  },
})
```

## Deno

```ts
import { telefuncWebSocket } from 'telefunc/websocket/deno'

const ws = telefuncWebSocket()

Deno.serve({ port: 3000 }, (request, info) => {
  const upgraded = ws.handleUpgrade(request, info)
  if (upgraded) return upgraded
  // ... rest of your handler
})
```

In Vite dev mode, Telefunc's Vite plugin sets this up automatically. You only need an adapter in production or when running outside Vite.

## Cloudflare Workers

```ts
import { telefuncWebSocket } from 'telefunc/websocket/cloudflare'

const ws = telefuncWebSocket()

export const TelefuncDurableObject = ws.createDurableObjectClass()

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    const resp = ws.handleTelefunc(request, env, ctx)
    if (resp) return resp
    // ... rest of your fetch handler
  },
}
```

## Delivery guarantees

### Message ordering

Messages within a channel keep their order.

### Buffering and recovery

<NoteWithoutIcon>
<b>Telefunc has built-in bounded recovery for active channels.</b>

1. Before the other side is connected and ready for the channel, new outgoing messages are buffered.

On the server, that buffer is capped by `channelSendBuffer`. When the channel is first created, the server waits up to `channelConnectTtl`. During a reconnect, it waits up to `reconnectTimeout`.

2. While the channel is connected, both sides keep a short replay history of what they already sent.

The client keeps `clientReplayBuffer`. The server keeps `serverReplayBuffer`.

3. If the connection drops, Telefunc detects that using `pingInterval`, reconnects, compares which channels are still alive, and replays whatever frames are still present in those replay histories.

Duplicate replayed frames are ignored.

4. Recovery succeeds only if the disconnect is short enough and the missing frames are still available.

If older data has already fallen out of the buffers, those messages are lost.

The channel closes with `ChannelNetworkError` only if Telefunc cannot recover the channel itself, for example when `reconnectTimeout` is exceeded or one side no longer acknowledges the channel after reconnect.

Buffered sends with acknowledgements reject when they are dropped.
</NoteWithoutIcon>

### Connection lifecycle

- The shared WebSocket opens on demand.
- After the last channel closes, it stays open for `idleTimeout` and then closes automatically.
- If the connection briefly drops, Telefunc tries to reconnect and resume the active channels.
- Temporary reconnects do not fire `onOpen()` or `onClose()`. Those callbacks are for the initial open and permanent close.

### Acknowledgements

Recovery handles short network interruptions, not application-level confirmation.

Use acknowledgements when the sender needs confirmation that the other side received and processed a message. See <Link href="/createChannel" />.

## Options

All adapters accept the following options:

| Option | Default | Description |
|---|---|---|
| `reconnectTimeout` | `60_000` | How long (ms) to hold channels open after a client disconnects, waiting for a reconnect. |
| `idleTimeout` | `60_000` | How long (ms) to keep the WebSocket open after the last channel closes. |
| `pingInterval` | `5_000` | How often (ms) the client sends a ping. The server considers the client dead after 2× this interval. Minimum: 1000ms. |
| `serverReplayBuffer` | `262_144` | Per-channel replay buffer size (bytes) on the server. Frames within this budget are replayed to a reconnecting client. |
| `clientReplayBuffer` | `1_048_576` | Per-channel replay buffer size (bytes) on the client. Frames within this budget are replayed to the server on reconnect. |
| `channelConnectTtl` | `5_000` | How long (ms) the server waits for the client to connect to a newly created channel. If the client does not connect within this window, the channel is closed automatically. |
| `channelSendBuffer` | `524_288` | Per-channel buffer size (bytes) for server-side messages while the client is not currently connected to the channel. Applies both when the channel is first created and during reconnect gaps. |
