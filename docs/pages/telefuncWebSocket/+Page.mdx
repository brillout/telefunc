Low-level WebSocket upgrade handler. Use this only if you need standalone WebSocket support without the runtime adapter.

## Node.js

```ts
import { telefuncWebSocket } from 'telefunc/websocket/node'
import { createServer } from 'node:http'

const server = createServer()
const ws = telefuncWebSocket()
ws.install(server)
```

Works with any Node.js HTTP server such as Express, Fastify, or Hono with `@hono/node-server`.

`install()` is idempotent, so calling it more than once on the same server is safe.

## Bun

```ts
import { telefuncWebSocket } from 'telefunc/websocket/bun'

const ws = telefuncWebSocket()

Bun.serve({
  port: 3000,
  websocket: ws.websocket,
  fetch(request, server) {
    const upgraded = ws.handleUpgrade(request, server)
    if (upgraded) return upgraded
    // ... rest of your fetch handler
  },
})
```

## Delivery guarantees

### Message ordering

Messages within a channel keep their order.

### Buffering and recovery

<NoteWithoutIcon>
<b>Telefunc has built-in bounded recovery for active channels.</b>

1. Before the other side is connected and ready for the channel, new outgoing messages are buffered.

On the server, that buffer is capped by `config.channel.bufferLimit`. When the channel is first created, the server waits up to `config.channel.connectTtl`. During a reconnect, it waits up to `config.channel.reconnectTimeout`.

2. While the channel is connected, both sides keep a short replay history of what they already sent.

The client keeps `config.channel.clientReplayBuffer`. The server keeps `config.channel.serverReplayBuffer`.

3. If the connection drops, Telefunc detects that using `config.channel.pingInterval`, reconnects, compares which channels are still alive, and replays whatever frames are still present in those replay histories.

Duplicate replayed frames are ignored.

4. Recovery succeeds only if the disconnect is short enough and the missing frames are still available.

If older data has already fallen out of the buffers, those messages are lost.

The channel closes with `ChannelNetworkError` only if Telefunc cannot recover the channel itself, for example when `config.channel.reconnectTimeout` is exceeded or one side no longer acknowledges the channel after reconnect.

Buffered sends with acknowledgements reject when they are dropped.
</NoteWithoutIcon>

### Connection lifecycle

- The shared WebSocket opens on demand.
- After the last channel closes, it stays open for `config.channel.idleTimeout` and then closes automatically.
- If the connection briefly drops, Telefunc tries to reconnect and resume the active channels.
- Temporary reconnects do not fire `onOpen()` or `onClose()`. Those callbacks are for the initial open and permanent close.

### Acknowledgements

Recovery handles short network interruptions, not application-level confirmation.

Use acknowledgements when the sender needs confirmation that the other side received and processed a message. See <Link href="/createChannel" />.

## Channel config

Set shared channel runtime settings on the server with `config.channel`:

| Option | Default | Description |
|---|---|---|
| `transports` | `['sse']`* | Channel transports: `['sse']`, `['sse', 'ws']`, or `['ws']`. |
| `reconnectTimeout` | `60_000` | How long (ms) to hold channels open after a client disconnects, waiting for a reconnect. |
| `idleTimeout` | `60_000` | How long (ms) to keep the shared connection open after the last channel closes. |
| `pingInterval` | `5_000` | How often (ms) the client sends a ping. The server considers the client dead after 2× this interval. Minimum: 1000ms. |
| `serverReplayBuffer` | `262_144` | Per-channel replay buffer size on the server. Frames within this budget are replayed to a reconnecting client. |
| `clientReplayBuffer` | `1_048_576` | Per-channel replay buffer size on the client. Frames within this budget are replayed to the server on reconnect. |
| `connectTtl` | `5_000` | How long (ms) the server waits for the client to connect to a newly created channel. If the client does not connect within this window, the channel is closed automatically. |
| `bufferLimit` | `524_288` | Per-channel buffer size for server-side messages while the client is not currently connected to the channel. Applies both when the channel is first created and during reconnect gaps. |

```ts
// Environment: server

import { config } from 'telefunc'

config.channel = {
  transports: ['ws'],
  reconnectTimeout: 60_000,
  idleTimeout: 60_000,
  pingInterval: 5_000,
  serverReplayBuffer: 262_144,
  clientReplayBuffer: 1_048_576,
  connectTtl: 5_000,
  bufferLimit: 524_288,
}
```

`*` When <Link text={<code>telefuncWebSocket()</code>} href="/telefuncWebSocket" /> is installed, the server enables both SSE and WS. The client starts on SSE and seamlessly upgrades to WebSocket in the background.
